Welcome to PartyPeople!

Though we might have a silly name, we have a mighty goal. There are enough barriers to voting as it is, so to make the process a little bit easier, PartyPeople provides information about current elections and candidates, as well as information about the political parties and political information about the states in the US.

While this site is intended for eligible voters to increase their knowledge about current elections in their area, at both a local, state, and federal level, anyone is welcome to browse for political information. You can go from one page to another, seeing how elections, candidates, parties, and states relate, and increasing your knowledge about the current political arena.

Data

Data Collection

For the first phase of the project, since only a few instances of each model were needed, team members manually collected data, either by using the API's listed below, or going to reputable sites like a candidate's personal website or a state's government website.

The second phase composed of a much greater momentum of data collection. The goal of the second phase was not only to make the pages dynamic, but to make the website insightful in some manner (more or less assembling lots of relevant information). In order to accomplish this task, a variety of steps needed to take place.

Locating Resourceful APIs

Of course, we can't just make up the data ourselves, so we have to look towards other publicly disclosed APIs that provide the data we need. While there are thousands of APIs out there, the APIs that really stick out are Google Civic API, GovTrackUs API, OpenFEC API, Huffington Post API, Mediawiki API, and Open States. Normally, it wouldn't be ideal to use so many APIs, but the fact of the matter is that the data we're looking for isn't condensed in a select few APIs.

Send RESTful Requests to APIs

You would think that the best part of data collection is finding the data sources, but assembling and interpreting the received data brings out your true inner excitement (but not the giggling kind of excitement, this is serious work). Much like using power tools in construction, we often use "tools" in programming, in this case, Python libraries/modules.

BeautifulSoup is perhaps one of our most widely used Python module in our scraping programs. It has a profound nature of translating a URL request into a multi-depth, dictionary based Python object. This module equips the programmer to easily map objects from the response to manipulate. The issue with using BeautifulSoup, however, is that BeautifulSoup is primarily for converting HTML into Python based objects (scraping website UIs).

Which brings me to the more important libraries/modules...

For generating and sending a request, urlopen is the recommended Python module to use for RESTful requests. Because urlopen is primarily based on the REST, a core feature that urlopen provides is the manner in which it manages responses. We use urlopen to send requests to the API endpoints, and to receiving the response from the API endpoints. Most of our scraping programs are written to feed urlopen as an argument into a BeautifulSoup object so we can continue to manipulate the data in the response. Now the question is, does Python provide a module for manipulating JSON specific responses?

Yes, they do indeed. Well it turns out the json, the Python module, translates the JSON response into a multi-depth dictionary based Python object, much like BeautifulSoup (is that a coincidence?). We use json to read the data in the json object in order to filter, massage, and interpret the received data.

When there are thousands (tens of thousands for us) of generated requests, the program starts to run slow. This is when the threading Python module comes in handy. Threading is a powerful computer science paradigm thats been around for decades, but threading can have grave consequences. You have to be careful about when you read or write to files that multiple threads may have access to. If you can manage these accesses in sync, then the beauties of multithreading will show in your program's performance.

Filter Data in Response

Its really simple to filter the data if you have the right tools. In our case, Python and its modules are very handy tools in making our task easier. In this stage, we stored the relevant data that we wanted to keep, and threw away any other data. Essentially, any data that pertained to political candidates, states, state elections, and political parties (especially any metadata that linked each of these) we tried to save locally.

Several general attributes of interest: foreign key ids, names, and critical meta data (such as location, etc.)

Massage Data

Finally, we have to "massage" (means to alter the data to your needs) the data to match the needs of the backend and frontend teams. This stage is important in ensuring that the data can be and is useful to these teams, else it is just a bunch of raw, senseless data. Also, this stage doesn't take place until we've collected and saved all the data from the listed data sources.

The steps we took in this stage: Update the files to ensure everything is in JSON format Map the saved data responses to each other Join the saved data responses Reorganize and massage the result of the join Forward final result to Backend (for database storage) and Frontend (for a json schema/example) teams.

For the second phase of the project, the following APIs were used to collect data to obtain many instances of each model:

Google Civic API: Candidates (names, party affiliation, elections they're in), Elections (basic information)
Open States API: all candidates in a State
BallotPedia: Elections(type, state it's in, description, election date), Candidates (names, summary, date of birth, party affiliation, previous offices held)
MediaWiki API: States (governors, basic State information), Parties (basic party information)
OpenFEC API: Federal Elections information and date, Candidate information
Huffington Post API: Polling information

Angular2 was added as the Front End framework for Phase 2. The benefits to using Angular is that the team could create dynamic web pages instead of hard coding a bunch of static pages. Angular has this concept of components, where you can tie HTML, CSS, and some TypeScript business logic all together into one directive. Two components were created for each model, one to display the models page of that model (a table with all the instances), and one component to display the details of an instance of that model. Components were also created for the header, the about page, and the home page.

Since Angular is dynamic one-page website, the team created a page with the header, and then inserted what's called a tag. Whenever the user goes to a URL, Angular will parse the route and insert the appropriate component into the tag, based on the routes the team previously defined.
